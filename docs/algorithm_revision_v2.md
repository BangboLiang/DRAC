# DRAC Algorithm Revision v2 Specification

## 1. Scope, evidence, and preserved baseline

This revision changes algorithms, tests, evaluation code, and generated results.
It does not edit the paper.  All v2 performance data must be generated by the
simulator or an explicitly identified measurement input.  No NIC directional
counter input is currently available, so v2 does not generate a profiler paper
figure.

The complete v1 result tree is preserved under
`results/archive/evaluation_v1/`, and the v1 report is preserved as
`reports/archive/final_change_summary_v1.md`.  The pre-repair eight-port baseline
is:

| Workload | DRAC-v1 | Sym-OCS |
|---|---:|---:|
| DP | 1.464 ms | 2.842 ms |
| PP | 7.452 ms | 5.952 ms |
| DP+PP Mixed | 8.585 ms | 6.632 ms |

In the v1 sparse-realization experiment, DRACSparse remained at approximately
24 stable directional channels across most epsilon values.  This missing
performance--resource trade-off is a required regression baseline, not a result
to erase.

## 2. Candidate-segment continuous target

For every contiguous candidate segment `[s,e]`, v2 solves one shared continuous
allocation directly:

\[
  Y^{*[s,e]} = \arg\min_{Y\in\mathcal F_{cont}}
  \sum_{k=s}^{e}L_k(Y),
\]

where

\[
  L_k(Y)=\max_{u,v:T^{(k)}_{uv}>0}
  \frac{T^{(k)}_{uv}}{B^{fix}_{uv}+cY_{uv}}.
\]

The epigraph implementation introduces one `theta_k` per communication node and
minimizes `sum(theta_k)` subject to:

- `Y_uv >= 0` and unreachable/diagonal allocations equal zero;
- `sum_v Y_uv <= n_tx_u`;
- `sum_u Y_uv <= n_rx_v`;
- `B_fix_uv + c Y_uv >= T_uv^(k) / theta_k` for every positive demand;
- `theta_k > 0`.

The constraints define a convex feasible set.  The implementation uses a
deterministic numerical convex optimizer, feasible warm starts derived from
single-node targets, explicit tolerances, and post-solve checks of every
capacity and completion-time constraint.  A numerically invalid result is never
reported as an optimum.

For `B_fix=0`, every demanded direction must receive positive continuous
capacity.  For `B_fix>0`, zero OCS allocation is permitted where fixed capacity
alone satisfies the epigraph constraint.

The segment logical cost is recomputed from the returned allocation rather than
trusted from optimizer metadata:

\[
  \widetilde C[s,e]=\sum_{k=s}^{e}L_k(Y^{*[s,e]}).
\]

All candidate targets, costs, solver status, feasibility residuals, and warm
start source are retained for debugging and result manifests.

## 3. Why medoid-only is insufficient

The v1 target dictionary restricts a segment to
`Y^(h), h in [s,e]`.  This is a strict subset of the continuous feasible region.
For a PP forward/backward pair, the two dictionary entries emphasize opposite
directions; the near-symmetric allocation that jointly minimizes their summed
drain time may be neither entry.  Thus v1 can choose correct boundaries while
still attach the wrong shared configuration.

The original dictionary implementation remains available as the
`MedoidTarget` ablation.  For every candidate segment, acceptance requires

`SegmentContinuousTarget cost <= MedoidTarget cost + numerical tolerance`.

The medoid is a feasible upper-bound candidate and a warm start, not a complete
segment oracle.

## 4. Symmetric target and fallback

For every candidate segment, v2 can also solve the same objective with
`Y_uv = Y_vu`.  The unrestricted directional feasible set contains the
symmetric feasible set, so the verified continuous directional cost must not
exceed the verified symmetric continuous cost.

Fallback is applied at two explicit levels:

1. **Segment level.** For each selected boundary interval, build and realize
   both unrestricted directional and symmetric candidates.  Compare actual
   integer segment costs.  A feasible lower-cost candidate wins; ties are broken
   by fewer units, then smaller endpoint peak footprint, then a deterministic
   lexicographic configuration key.  The selected type and reason are logged.
2. **Schedule level.** Construct complete directional, complete symmetric, and
   segment-fallback schedules.  Compare realized communication cost plus the
   identical reconfiguration-delay model.  Ties use the schedule-wide stable
   pool, total connection units, then a deterministic schedule key.

The schedule-level candidate set includes the same-budget Sym-OCS schedule.
Consequently the planner can claim simulator no-harm only relative to that exact
candidate under the same ordered sequence, physical budget, realization
tolerance, and delay model.  It is not a claim about unmeasured hardware.

## 5. Dynamic programming

For a chosen candidate-cost family, the recurrence remains:

\[
 OPT[e]=\min_{0\le q<e}\{OPT[q]+\widetilde C[q,e-1]
              +\mathbf 1[q>0]\delta\},\quad OPT[0]=0.
\]

The default v2 `SegmentOpt-DynamicProgramming` cost matrix comes from direct
segment reoptimization, not the medoid dictionary.  Separate matrices are kept
for directional, symmetric, and medoid ablations.  Every result stores:

- candidate target and verified communication cost;
- target type and numerical solver diagnostics;
- `previous[e]` predecessor values;
- backtracked boundaries;
- selected target type and fallback reason.

For small `K`, exhaustive enumeration of all `2^(K-1)` contiguous partitions
using the same candidate costs is the partition oracle.  It is distinct from a
medoid oracle.

## 6. Multi-seed sparse integer realization

The objective is unchanged:

\[
  \min_X\|X\|_1\quad\text{s.t.}\quad X\in\mathcal F_{OCS},
  \quad C_m(X)\le(1+\epsilon)\widetilde C_m.
\]

V2 evaluates at least two independent paths.

### 6.1 FloorSeed

Start from `floor(Y*)`, clip only unreachable entries, validate Tx/Rx/global
budgets, and add missing mandatory coverage where positive demand has no fixed
capacity.  This path is useful for strict tolerance.

### 6.2 SparseCoverageSeed

Start from zero.  Add one unit to every direction that has positive demand in
the segment and no fixed capacity.  If the mandatory coverage itself violates a
budget, return an explicit resource-constrained result and request map.  This
path does not inherit density from the continuous target.

Each seed stops additions as soon as the performance bound is met.  The final
`MultiSeed` result is selected from feasible seed results by fewer units, smaller
endpoint peak footprint, lower cost, and deterministic configuration order.

## 7. Bottleneck groups

For each node under the current integer configuration, v2 computes all demanded
directions whose drain times equal the node maximum within a scale-aware
tolerance.  If every useful single addition has zero gain, it constructs bounded
joint actions from these tied bottleneck directions.  A group action adds the
minimum one-unit set that is jointly Tx/Rx feasible and evaluates the true
segment-cost reduction.  Positive-gain groups compete by gain per added unit,
absolute gain, group size, and lexicographic order.

The algorithm does not terminate merely because the best single-direction gain
is zero while a feasible tied group has positive combined gain.

## 8. Swap local search

After reaching tolerance, and between pruning passes when useful, v2 examines a
bounded deterministic set of

`X - E_uv + E_ab`

actions.  A swap must preserve reachability, non-negativity, Tx/Rx/global
feasibility, serviceability, and the epsilon bound.  It is accepted if it:

1. reduces the endpoint peak-footprint proxy; or
2. keeps that proxy and unit count unchanged but lowers segment cost.

First-improvement is used with lexicographically ordered removal/addition pairs;
the search repeats until no improving swap exists or a documented iteration cap
is reached.  A subsequent pruning pass may convert a cost-improving equal-count
swap into a net unit reduction.  Swap count, pre/post cost, and footprint change
are logged.

## 9. Reverse pruning

For every removable unit, compute

\[
 \Lambda_{uv}=C_m(X-E_{uv})-C_m(X).
\]

Among removals that preserve serviceability, integer feasibility, and the
epsilon bound, delete the unit with smallest loss; ties use `(src,dst)`.  Repeat
until no valid deletion remains.  Pruning count and each accepted loss are
recorded.  The final cost is recomputed and revalidated.

## 10. Epsilon semantics and monotonicity

Tolerance is exactly

`C_m(X) <= (1 + epsilon) * C_tilde_m`

with scale-aware numerical slack used only for floating-point comparison.
Epsilon experiments run in ascending order.  Every feasible configuration from
a stricter epsilon is added unchanged to the next epsilon's candidate set and
revalidated against the wider bound.  If it wins, the result records historical
reuse.  This legal candidate reuse guarantees that reported connection count is
non-increasing without editing result CSVs.

Addition stops immediately when tolerance is reached; residual target mismatch
is not an objective.

## 11. Schedule-wide compaction interface

The physical planner formula remains unchanged.  It consumes the selected
integer configurations:

`nbar_tx[u] = max_m sum_v X_m[u,v]`,

`nbar_rx[u] = max_m sum_v X_m[v,u]`.

V2 retains independent Tx/Rx pools, bidirectional bundles, stable exposed
channels, physical bindings, and constrained-direction requests.  Compaction is
recomputed only after realization selection.  A peak-aware realization
ablation is considered only if segment units decrease but schedule-wide peaks do
not.

## 12. Correctness and acceptance criteria

### Segment target and segmentation

- every target satisfies Tx/Rx, reachability, and epigraph constraints;
- verified segment cost is no worse than its medoid feasible upper bound;
- unrestricted cost is no worse than symmetric continuous cost;
- PP forward/backward produces a new near-symmetric shared allocation;
- Dynamic Programming exactly matches exhaustive partition enumeration on small
  cases, including boundaries and cost;
- delta zero approaches fine-grained segmentation and very large delta yields
  OneConfig;
- repeated runs are deterministic;
- fallback decisions are explicit and reproducible.

### Integer realization

- every returned configuration passes integer Tx/Rx/global validation;
- every positive-demand direction is served by OCS or fixed capacity;
- every result marked feasible satisfies its epsilon bound;
- failures are marked resource-constrained and retain a request map;
- tied-bottleneck tests make progress through group actions;
- at least one constructed test demonstrates useful swap behavior;
- reverse pruning never violates tolerance;
- MultiSeed matches or approaches the small exhaustive minimum-unit oracle;
- connection counts are non-increasing across ascending epsilon;
- DRACSparse uses fewer units than FillAllResidual for at least one accepted
  epsilon/workload case.

### System experiments

- PP and Mixed retain all negative outcomes and compare against archived v1;
- final fallback schedule cannot exceed the included Sym-OCS simulator candidate
  under identical assumptions;
- an epsilon trade-off must be present before generating the final v2 realization
  figure; otherwise full evaluation stops for diagnosis;
- fixed-budget compaction and iso-performance minimum-pool results are reported
  separately;
- all plots read persisted CSV/JSON data and have both PDF and PNG output.
